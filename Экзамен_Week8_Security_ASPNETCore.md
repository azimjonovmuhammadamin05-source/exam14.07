# Экзаменационное задание — Week 8: Security with ASP.NET Core
**Формат:** индивидуальное практическое задание (take-home)
**Время на выполнение:** 24 часа с момента получения задания
**Технологии:** ASP.NET Core Web API, ASP.NET Core Identity, JWT, Entity Framework Core

---

## Легенда / Постановка задачи

Вы — backend-разработчик в стартапе, который создаёт платформу **"TaskHub"** — сервис управления задачами для команд. Продукту нужна безопасная система пользователей: регистрация, вход, восстановление пароля, роли (обычный пользователь / менеджер / администратор) и защищённые API-эндпоинты.

Ваша задача — реализовать backend-часть системы безопасности с нуля, используя ASP.NET Core Identity и JWT-аутентификацию.

---

## Технические требования

### Базовая модель предметной области
- `ApplicationUser` (наследник `IdentityUser`) — минимум с полями `FullName`, `CreatedAt`
- `TaskItem` — сущность задачи: `Id`, `Title`, `Description`, `IsCompleted`, `OwnerId` (связь с пользователем)

### Часть 1. Аутентификация и авторизация (основы)
- Настроить проект ASP.NET Core Web API
- Продумать и кратко обосновать в README разницу между Authentication и Authorization в контексте вашего приложения
- Настроить middleware pipeline в правильном порядке (`UseAuthentication` → `UseAuthorization`)

### Часть 2. ASP.NET Core Identity
- Подключить и настроить `ASP.NET Core Identity` с Entity Framework Core (SQLite или SQL Server — на выбор)
- Настроить политику паролей (минимальная длина, требование цифр/спецсимволов и т.д.)
- Эндпоинты:
  - `POST /api/auth/register` — регистрация нового пользователя
  - `POST /api/auth/login` — вход (email + пароль)
  - `POST /api/auth/logout` — выход
- Убедиться, что пароли хранятся только в виде хэша (объяснить в README, какой алгоритм хэширования использует Identity)

### Часть 3. JWT-аутентификация
- При успешном логине выдавать JWT access-токен
- В токен должны быть встроены claims: `UserId`, `Email`, `Role`
- Настроить валидацию токена (issuer, audience, lifetime, signing key) через конфигурацию (`appsettings.json`, не хардкодить секреты в коде)
- Защитить эндпоинты работы с задачами (`/api/tasks/*`) атрибутом `[Authorize]`
- Реализовать CRUD для `TaskItem`, где пользователь видит и редактирует **только свои** задачи (проверка через claims из токена)

### Часть 4. Расширенные функции Identity
- Реализовать flow **Forgot Password**:
  - `POST /api/auth/forgot-password` — генерация reset-токена и "отправка" письма
  - Письмо можно не отправлять реально — обязательно реализовать интерфейс `IEmailSender` и заглушку, которая **логирует содержимое письма в консоль/файл** (эмуляция SMTP). Указать в README, как выглядела бы реальная SMTP-конфигурация (host, port, credentials)
- Реализовать flow **Reset Password**:
  - `POST /api/auth/reset-password` — установка нового пароля по токену
- **Бонус (не обязательно, доп. баллы):** внешний вход через Google OAuth (`AddGoogle` в Identity). Если не успеваете — опишите в README, как бы вы это реализовали (шаги + код-скелет).

### Часть 5. Ролевая авторизация
- Создать минимум 3 роли: `User`, `Manager`, `Admin`
- При регистрации по умолчанию присваивать роль `User`
- Эндпоинт (доступен только `Admin`) для назначения роли другому пользователю:
  - `POST /api/admin/assign-role`
- Защитить хотя бы 2 эндпоинта через `[Authorize(Roles = "...")]`
- Реализовать хотя бы одну проверку авторизации на основе **claims** (не ролей) — например, через `[Authorize(Policy = "...")]` и кастомный `AuthorizationHandler` (например, политика "может редактировать только свою задачу или является Manager")
